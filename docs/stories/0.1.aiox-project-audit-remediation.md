## Story
**As a** maintainer do projeto OpenF1 Telemetry Platform,
**I want** uma auditoria técnica guiada por AIOX com correções para falhas reais,
**so that** o repositório volte a atender os gates básicos de qualidade e a integração frontend/backend funcione de forma consistente.

## Acceptance Criteria
1. Os gates da raiz executam sem falhas de recursão do Turborepo.
2. A suíte principal de testes do backend roda em `pytest` sem erro estrutural.
3. O frontend usa os endpoints e formatos reais retornados pela API para sessões, pilotos e posições.
4. Paths críticos do backend não dependem do diretório atual de execução.
5. A revisão registra achados, correções aplicadas, riscos residuais e referências de documentação oficial consultadas.

### Completion Notes
Os gates recursivos da raiz, a suíte `pytest` do backend, a resolução de paths dependente de `cwd`, o heartbeat e a resolução de pilotos do WebSocket foram corrigidos. O frontend também foi alinhado aos contratos reais da API para sessões, telemetry, weather e predictions.

O mapeamento AIOX identificou que a navegação prometia páginas inexistentes. Isso foi fechado com a criação das rotas `/telemetry`, `/drivers`, `/circuits`, `/historical`, `/settings` e `/help`, além da revisão visual do shell da aplicação, header/sidebar, estados vazios, superfícies reutilizáveis, landing page e filtros principais.

As previsões deixaram de depender de mocks e passaram a montar contexto real da sessão via OpenF1. O backend agora expõe o status real do motor de predição, distinguindo fluxo com modelo treinado, modo híbrido e fallback heurístico. Como os artefatos treinados não estão presentes em `packages/ml-models`, o comportamento atual é majoritariamente heurístico com dados reais de corrida.

No backend de dados históricos, a integração F1DB foi estabilizada com correção de acesso a `sqlite3.Row`, normalização de formatos de classificação e testes de contrato. A integração FastF1 foi corrigida com cache de sessão, leitura correta de metadata, saneamento de `NaN` em respostas de laps e carregamento progressivo no frontend histórico.

O smoke test manual validou `dashboard`, `telemetry`, `predictions`, `drivers`, `circuits`, `historical`, `settings` e `help` em `localhost`. O fluxo histórico passou a responder com `200` nas rotas FastF1 principais e a hidratar o workspace progressivamente sem `500` ou erro de runtime, ainda que mais lentamente que as páginas live.

Riscos residuais: a OpenF1 free tier continua sendo o principal limitador para produção, ainda existem warnings de lint herdados no frontend e o workspace histórico segue mais pesado que as páginas live por depender do pipeline FastF1.
